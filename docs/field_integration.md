# 现场接入与安全门槛

## 局部导航接口只读确认

在机器人已配置 ROS Master 的终端运行 `scripts/ros1_readonly_inspect.sh`，保存完整输出。再针对实际话题/节点执行：

```bash
rostopic hz <实际话题>
rostopic echo -n 1 <实际话题>
rosnode info <实际导航节点>
rosservice type <实际服务>
```

禁止用 `rostopic pub` 测试 `/cmd_vel`。主线不检查或加载 `/map`，重点确认 `/odom`、`/scan`、TF、平台安全状态，以及成熟局部控制器/RCS 能否接收 `odom` 局部目标、查询状态和停止。`move_base` 只有在平台明确配置为滚动局部 costmap、接受 `odom` 短距离目标且不依赖静态全局地图时才可作为可选适配器；只有裸速度接口时维持 Mock。

本机为 Ubuntu 22.04，而图漾/ROS Noetic 官方组合为 Ubuntu 20.04。Camport 原生构建若遇到 PCL 1.10/1.12 ABI 冲突，使用 `scripts/build_camport_container.sh` 构建隔离的 Noetic Focal 相机环境；不得通过伪造系统头文件软链接绕过 ABI 检查。

## 分阶段接入

1. 用录制 RGB-D 离线回放。
2. 静止连接 ROS，只订阅相机、里程计、实时激光和 TF，不加载全局地图。
3. RViz 只显示 odom 局部目标、滚动局部代价图和短路径，不执行运动。
4. 验证导航取消、急停和人工接管。
5. 获批后在空旷区域以平台批准的最低速度执行 0.3—0.5 m 的单段局部目标。
6. 验证静态障碍、无局部路径、数据超时、里程计丢失和目标丢失。
7. 最后运行“目标已可见”的短距离文字寻物闭环。

任一步失败都退回上一阶段。未确认成熟导航接口或安全批准时，交付真实数据回放、Mock 导航和路径可视化。

## RGB-D 现场记录项

- Percipio Viewer、固件、SDK/驱动版本和负责人/日期
- 原始 RGB、Depth、CameraInfo、点云样例及允许使用范围
- 分辨率、帧率、深度单位/无效值、RGB-D 对齐和最大时间差
- `camera_link` 坐标轴方向、`camera_link → base_link` 实测外参
- `odom → base_link → camera_link` 短时坐标树、里程计新鲜度与时间同步方式

## 无地图边界

- 运动前目标必须已在当前视野连续确认；无目标不自主前进、不访问观察点。
- 局部窗口建议从 8 m × 8 m、0.05 m 分辨率起步，真实值须依据传感器视野和算力实测。
- 深度/点云和实时激光持续更新滚动局部图，障碍进行机器人外形与安全余量膨胀。
- 每执行约 0.3—0.5 m 重新检测目标、更新地图、生成停靠点并重规划。
- 目标移出视野、局部路径被长墙等障碍截断或窗口内无解时停止，不猜测窗口外路线。

未实测前，这些参数保持“未知”，不得从模拟配置反推真实设备规格。
